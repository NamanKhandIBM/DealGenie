## 7. Overview of NS1 Connect Pricing Metrics

Two primary metrics determine the cost of delivering NS1 Connect services:

**Network size** — The number of domains managed by a customer is a key factor in determining network resources needed to manage DNS traffic.

**Network traffic** — The amount of internet traffic a customer experiences dictates how much compute NS1 needs to respond to DNS queries.

To price NS1 Connect, two proxies are used in tandem:

- **Network size = DNS records.** Charged by the maximum number of DNS records a customer wants. Hybrid Cloud DNS bundles feature pre-set numbers of records; NS1 Connect allows manual selection.
- **Network traffic = DNS queries per month (QPM).** Charged by the maximum number of DNS queries expected per month.

---

## 8. Quirks of NS1 Connect Quoting Metrics

To make the NS1 SaaS pricing model work within IBM's quoting framework (CPQ), a few quirks were introduced. The metrics CPQ Config outputs on a quote may look different from what's entered in the form.

**Baseline conversion metrics:**
- 1 IBM "Record" = 1,000 DNS records
- 1 IBM "Request" = 10 million queries per month (baseline queries)
- 1 IBM "Interaction" = 1 million queries per month (for RUM-based traffic steering queries)

---

## 9. Tiered Usage Pricing Overview

- NS1 Connect uses a tiered system: more queries/records purchased = cheaper per-unit pricing as you move up the tier chart.
- A quote's usage figure places the customer in a usage pricing tier based on the quantity of each metric.
- Overage charges apply if a customer exceeds contracted usage for queries or records in a given month.
- Overage charges can be (and often are) waived for seasonal spikes or one-time events beyond a customer's control.
- Consistent overages = upsell opportunity to a higher usage tier to avoid recurring overage fees.

## 11. Quoting Hybrid Cloud DNS (the Bundles) in CPQ

### Step 1 — Two Easy Steps to Price Hybrid Cloud DNS
*(Gather this info before entering CPQ)*

**First — pick a bundle based on number of authoritative DNS records used by the customer:**
- Under 200K records → **"Enterprise"** (Part **D0GYUZX**) — most IBM customers fit here
- 200K – 2M records → **"Enterprise Plus"** (Part **D0GYWZX**) — for sophisticated/specialized use cases
- 2M+ records → use the a la carte NS1 Connect option instead

**Second — set the price using number of requests:**
- Each request = 10 million queries per month (QPM)

> Don't forget to add services!

*Note: other bundle elements (filter chains, monitors, etc.) may matter, but records is the primary determining factor for most customers.*

### Step 2 — Two Easy Steps to Price Upsell GSLB
*(Gather this info before entering CPQ)*

**First** — Ask the customer if they'll use NS1's Real User Monitoring (RUM) data, or bring their own:
- Using NS1's data = **GSLB Standard** (Part **D0GZ0ZX**)
- Customer wants to configure/use private RUM tests = **GSLB Advanced** (Part **D0GYYZX**)

**Second** — Determine the number of interactions to be processed using GSLB:
- Each interaction = 1 million queries using RUM data
- Sold in 5-million-query (5-interaction) packages

> Don't forget to add services!

### Steps to Generate a New Customer Quote in CPQ

**GSLB / RUM-based steering:**
1. Specify if IBM needs to configure RUM data from the customer's private sources (if not, pre-configured NS1 RUM data is used)
2. Input number of RUM-based queries per month (CPQ auto-generates "interactions"). Must be a multiple of 5. Minimum is 5M QPM for private RUM sources, 1M QPM for NS1 data.

**Hybrid Cloud DNS:**
1. Answer "no" to the entitlement question
2. Choose the correct bundle based on number of records the customer wants
3. Enter the queries per month (CPQ auto-generates "requests")

**And then:**
1. Choose the contract term (in months, min 12 / max 60)
2. Choose the subscription billing cycle

## 12. Quoting NS1 Connect Premium in CPQ

### Steps to Generate an NS1 Connect Quote in CPQ

**Header section:**
1. Answer "no" to the entitlement question
2. Select the add-ons/upsells to quote (Managed DNS shows by default — it's a prerequisite for other solutions)
   - Add-on/upsell options: Dedicated DNS, DNS for China, RUM-based traffic steering (note: "regular" filter chains are sold through Managed DNS), DNS Insights

**Managed DNS section:**
1. Enter DNS queries per month in millions — auto-converts to "requests" (1 request = 10M queries). 10M query minimum.
2. Enter total DNS records in thousands — auto-converts to "records" (1 record = 1,000 NS1 records). 1K record minimum.
3. Enter number of monitors (1 monitor = 1 monitoring "job" = 1 "DNS job")
4. Enter number of filter chains to buy (1 filter chain = 1 "resource unit")
5. Check additional boxes as needed: DDoS overage protection, NXDOMAIN waivers, enhanced monitor intervals, vanity name servers

**Dedicated DNS section:**
1. Enter number of Points of Presence (PoPs) for the dedicated network. Minimum 3, maximum 12.

**DNS for China section:**
1. Enter the number of DNS records to be used in mainland China specifically, in millions of queries per month (queries not handled by the regular anycast NS1 Connect network). Auto-converts to "requests" (1 request = 10M queries).
   - *Only appears if the relevant box was checked before Managed DNS.*

**RUM traffic steering section:**
1. Specify if IBM needs to configure RUM data from the customer's private sources (otherwise pre-configured NS1 data is used)
2. Enter queries per month (in millions) routed through RUM-based filter chains — a subset of total Managed DNS query count. Auto-converts to "interactions" (1M RUM queries = 1 interaction). Minimum 1 million queries.

**DNS Insights section:**
1. Check the "include DNS Insights" box — automatically includes the upsell amount based on query counts entered previously
2. Optionally enter the number of custom DNS Insights policies wanted — no minimum
   - *Only appears if the relevant box was checked before Managed DNS.*

**Term and billing tab:**
1. Select contract term in months — minimum 12, maximum 60
2. Select billing frequency

> Don't forget to add services!

## 14. Appendix B: Upsell Overview — GSLB

**Customer benefits:**
- **Performance:** Optimized end-user experience based on actual backend service performance (CDNs, clouds) on real user devices (not synthetic data)
- **End-to-end visibility:** Load balancing all the way to end-user devices
- **Flexibility:** Use NS1's RUM data (standard) or bring your own RUM data sources (advanced)
- **Lower cost:** Dramatically cheaper than traditional load balancing offerings

Pre-packaged bundles include basic traffic steering, but many customers want more sophistication using RUM data for GSLB.

**Two upsell packages:**
- **GSLB Standard** — set number of queries using NS1's RUM data — Part **D0GZ0ZX**
- **GSLB Advanced** — set number of queries using customer's own RUM data feeds — Part **D0GYYZX**

### GSLB Upsell Details

**GSLB Standard — Part D0GZ0ZX**
- Price based on number of GSLB queries (minimum 1B queries for this package)
- Plus: 100 filter chains, NS1-provided RUM data package
- Sold in 1M query-per-month blocks

**GSLB Advanced — Part D0GYYZX**
- Price based on number of GSLB queries (minimum 5B queries for this package)
- Plus: 100 filter chains, Private data package (custom ingestion of customer-provided data feed)
- Sold in 5M query-per-month blocks

## 18. Appendix F: Metric Descriptions

### Legalese Edition

**Request** — The act by the Client authorizing IBM to perform a service submitted to or managed by the Cloud Services. For this Cloud Service, a Request = 10 million queries which are requests for authoritative DNS information made to an IBM NS1 Connect Managed DNS or IBM NS1 Connect Dedicated DNS nameserver.

**Interaction** — A digital interaction with an application captured by the Cloud Service. For this Cloud Service, an Interaction = 1 million DNS queries for which RUM (real user metrics) data can be used to determine the best possible outcome.
